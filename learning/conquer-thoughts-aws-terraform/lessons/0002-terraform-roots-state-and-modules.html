Terraform Roots, State, and Modules

Lesson 0002 - Conquer Thoughts AWS/Terraform - 2026-06-14

Part of your mission to build a durable mental model of the Conquer Thoughts production walking skeleton so you can explain, debug, and safely extend the AWS/Terraform/GitHub Actions rail.

After this lesson you'll be able to explain why the repo has `account-bootstrap`, `bootstrap`, `production`, and reusable modules.

Concrete example: the chicken-and-egg root

The deploy pipeline needs an IAM role before it can call AWS. Terraform also needs somewhere safe to store state before teams or workflows can collaborate. That is why infra/terraform/bootstrap exists: it creates the state bucket, the deploy and plan roles, state-access policies, and the ECR repository that the first image push needs.

Terraform state is not a log of commands; it is Terraform's binding between configuration addresses and real remote objects. HashiCorp describes state as the mapping Terraform uses to connect resource instances in configuration to real infrastructure objects (Terraform state docs). If that mapping is lost or split across laptops, Terraform cannot safely reason about what it owns.

Concrete example: the account singleton

AWS allows the GitHub OIDC provider for token.actions.githubusercontent.com to be an account-level trust object. The project extracted it into infra/terraform/account-bootstrap so rehearsal and production bootstrap roots do not fight over the same singleton. The helper script creates or imports that provider, then migrates its state to S3 after the bucket exists.

The bootstrap root now reads that provider as data and creates environment-shaped roles around it. The removed block in bootstrap is a state-management move: forget the old environment-owned provider address without destroying the real account singleton.

The principle

Terraform layout is ownership design. A root module is a lifecycle boundary: apply it, store its state, and decide who is allowed to run it. A child module is an abstraction boundary: compose a higher-level architectural thing from lower-level provider resources. HashiCorp's module guidance says good modules raise the level of abstraction and warns against thin wrappers around single resources (Terraform module docs).

Remote state and lockfiles

The S3 backend stores state at a key inside a bucket and supports state locking when use_lockfile = true. HashiCorp also recommends enabling bucket versioning so state can be recovered after accidental deletion or human error (Terraform S3 backend docs). That is why the project state bucket has versioning, encryption, public access blocking, and native lockfiles.

Check yourself

  1. Which root owns the GitHub OIDC provider singleton?
    Answer

    account-bootstrap. It keeps the account-level provider out of environment teardown.

  2. Why does bootstrap exist instead of letting the GitHub deploy workflow create everything?
    Answer

    The workflow needs an IAM role, ECR repository, and remote state before its first deploy can safely run.

  3. What is the difference between a root module and a child module in this repo?
    Answer

    A root module is applied and stateful as its own lifecycle. A child module is called by a root to package reusable infrastructure concepts.

  4. Why is state migration from local to S3 not optional housekeeping?
    Answer

    If state stays local, future applies, destroys, or imports cannot reliably see what Terraform already owns.

  5. A future PR wants to add a module that wraps exactly one aws_s3_bucket. What should you ask?
    Answer

    Ask what architectural concept the module names. If it is only a thin wrapper with no higher-level meaning, direct resource use may be clearer.
