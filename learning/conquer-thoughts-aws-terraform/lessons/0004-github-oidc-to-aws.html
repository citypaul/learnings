GitHub OIDC to AWS

Lesson 0004 - Conquer Thoughts AWS/Terraform - 2026-06-14

Part of your mission to build a durable mental model of the Conquer Thoughts production walking skeleton so you can explain, debug, and safely extend the AWS/Terraform/GitHub Actions rail.

After this lesson you'll be able to draw the OIDC token exchange from GitHub workflow to AWS STS credentials.

Concrete example: the workflow has no AWS access key

The deploy workflow has permissions: id-token: write and contents: read. GitHub's AWS OIDC guide says id-token: write lets the workflow request a JSON Web Token; it does not itself grant resource write permission (GitHub OIDC in AWS docs).

The workflow then runs aws-actions/configure-aws-credentials with a role ARN, region, account allow-list, and session name. That action asks GitHub for a signed identity token and asks AWS STS to exchange it for temporary AWS credentials. No long-lived AWS_ACCESS_KEY_ID or AWS_SECRET_ACCESS_KEY is stored in GitHub.

Concrete example: AWS has a front door for GitHub

AWS needs to know that it should trust tokens issued by GitHub. The account-level IAM OIDC provider represents https://token.actions.githubusercontent.com, and AWS's IAM docs describe OIDC identity providers as the object that establishes trust between an external issuer and an AWS account (AWS OIDC provider docs).

The missing piece: there is no shared secret

AWS does not verify GitHub's JWT with a password or a shared secret. GitHub signs the token with a private signing key. AWS discovers GitHub's public keys from GitHub's OIDC metadata and uses the matching public key to verify the signature. AWS's OIDC provider docs require the discovery document to include jwks_uri, the endpoint where IAM gets public keys used to verify signed tokens (AWS OIDC provider docs).

GitHub publishes that discovery document at https://token.actions.githubusercontent.com/.well-known/openid-configuration. OpenID Connect Discovery defines this metadata pattern, including jwks_uri (OpenID Connect Discovery 1.0). The JWT header includes fields such as alg and kid; GitHub's OIDC reference describes kid as the token's key identifier, and the JWS spec says kid is a hint for which key secured the signed object (GitHub OIDC reference, RFC 7515: JWS). A signed JWT is not necessarily encrypted: AWS can read the claims, but it cannot forge or alter the token without breaking the signature.

Trusting GitHub as an issuer is only the first gate. The IAM role trust policy then checks which GitHub workflow identity may assume the role. For this project, the deploy role accepts: 

aud = sts.amazonaws.com
sub = repo:citypaul/conquer-thoughts:ref:refs/heads/main

The audience says "this token is intended for AWS STS." The subject says "this exact repository and branch identity may deploy." AWS's GitHub OIDC role docs require a constrained token.actions.githubusercontent.com:sub condition rather than a missing, null, or wildcard-only subject (AWS role for GitHub OIDC docs).

The principle

OIDC turns deployment identity into a short-lived proof instead of a stored secret. GitHub proves "this workflow run is from this repo/ref" by issuing a signed JWT. AWS first verifies the token's signature using GitHub's public keys, then checks the token claims against the role trust policy. Only then does AWS STS return temporary credentials, which the AWS CLI, Terraform, and Docker/ECR commands can use.

Reading and resources

  • GitHub: Configuring OIDC in AWS - the practical setup path for id-token: write, aud, sub, and the AWS credentials action.
  • GitHub OIDC reference - the claim and header reference, including iss, aud, sub, exp, and kid.
  • GitHub OIDC discovery document - the live metadata document AWS can use to discover GitHub's issuer metadata and public key set location.
  • AWS IAM OIDC provider docs - the key source for jwks_uri, TLS/thumbprint handling, and how AWS retrieves public keys.
  • OpenID Connect Discovery 1.0 - the protocol-level explanation of provider metadata and jwks_uri.
  • RFC 7515: JSON Web Signature and RFC 7517: JSON Web Key - read these when you want the cryptographic shape behind signed JWTs and public key sets.

Check yourself

  1. What does id-token: write let a GitHub job do?
    Answer

    Request a GitHub OIDC token for the run. It does not directly grant AWS permissions.

  2. Which AWS API turns the GitHub token into temporary credentials?
    Answer

    AssumeRoleWithWebIdentity through AWS STS. AWS documents that it returns temporary security credentials after validating the web identity token.

  3. How can AWS verify that GitHub signed the JWT without sharing a secret with GitHub?
    Answer

    GitHub signs the token with a private key. AWS discovers GitHub's public keys through the OIDC discovery/JWKS endpoint and uses the matching public key to verify the JWT signature.

  4. Why does the deploy trust check both aud and sub?
    Answer

    aud confirms the token is intended for AWS STS; sub scopes the identity to the exact repository and branch shape.

  5. If the workflow uses a GitHub deployment environment later, what must change?
    Answer

    The OIDC subject shape changes to an environment subject, so the IAM trust policy must be deliberately updated and the environment must have protection rules.

Go deeper: AWS IAM OIDC provider docs - read this for the exact discovery/JWKS mechanism AWS uses to find public keys for signed tokens.
