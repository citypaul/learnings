VPCs and CIDR Blocks

Lesson 0007 - Conquer Thoughts AWS/Terraform - 2026-06-14

Part of your mission to build a durable mental model of the Conquer Thoughts production walking skeleton so you can explain, debug, and safely extend the AWS/Terraform/GitHub Actions rail.

After this lesson you'll be able to explain what the Conquer Thoughts VPC owns and how its 10.42.0.0/16 address space is carved into smaller subnet ranges.

Concrete example: the VPC is the network container

In the network module, aws_vpc.this creates the private network that the rest of the AWS runtime is placed inside: 

resource "aws_vpc" "this" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true
  enable_dns_hostnames = true
}

The default vpc_cidr is 10.42.0.0/16. AWS describes a VPC as a logically isolated virtual network in an AWS Region, containing subnets, route tables, gateways, and other network controls (AWS VPC overview). So the VPC is not "the internet bit" or "the server bit." It is the address-and-routing container for all of those bits.

Concrete example: CIDR is the address budget

AWS requires an IPv4 CIDR block when creating a VPC, and the VPC CIDR block must be between /16 and /28. The AWS VPC CIDR docs also recommend using private IPv4 address ranges when the addresses should not be directly reachable from the public internet (AWS VPC CIDR docs).

The project chose 10.42.0.0/16, which gives one large private range that can be split into smaller subnet ranges. The walking skeleton currently spends four /24 slices of that budget:

SliceProject CIDRsPurpose
Public task subnets 10.42.0.0/24, 10.42.1.0/24 Fargate task placement with public IPv4 assignment for day-one egress.
Private origin subnets 10.42.10.0/24, 10.42.11.0/24 Internal ALB placement for the CloudFront VPC origin path.

The principle

A VPC is the regional network boundary. A CIDR block is the private address budget inside that boundary. Subnets are non-overlapping chunks of that budget, and each subnet lives in one Availability Zone. AWS's subnet docs make the important split explicit: the subnet CIDR must be inside the VPC CIDR, subnets cannot overlap, and a subnet is associated with one Availability Zone (AWS subnet docs).

Why the ranges matter

CIDR planning is boring until it is not. If ranges overlap, routing cannot be unambiguous. If ranges are too small, services can run out of IP addresses. AWS reserves the first four IP addresses and the last IP address in every subnet, so a /24 subnet has 251 usable addresses, not 256 (AWS subnet CIDR sizing docs).

For this project, the /16 VPC leaves plenty of room for later private app subnets, database subnets, VPC endpoints, or separate cells. The specific /24 subnets are small enough to read by eye and large enough for the current walking skeleton.

Check yourself

  1. What does 10.42.0.0/16 represent in the project?
    Answer

    The VPC's private IPv4 address budget. Subnets must be carved from inside that range.

  2. Is the VPC tied to one Availability Zone?
    Answer

    No. The VPC is regional. Each subnet is tied to one Availability Zone.

  3. Why can't 10.42.0.0/24 and 10.42.0.128/25 both be normal subnets in the same VPC?
    Answer

    They overlap. AWS requires subnet CIDRs in a VPC to be non-overlapping.

  4. Why does a /24 subnet not provide all 256 addresses to workloads?
    Answer

    AWS reserves five addresses in every subnet: the first four and the last address.

  5. What future work does a large VPC CIDR make easier?
    Answer

    Adding more subnet groups, such as private task subnets, database subnets, VPC endpoints, or additional cells, without renumbering the whole network.
