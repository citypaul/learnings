AWS Walking Skeleton Map

Lesson 0001 - Conquer Thoughts AWS/Terraform - 2026-06-14

Part of your mission to build a durable mental model of the Conquer Thoughts production walking skeleton so you can explain, debug, and safely extend the AWS/Terraform/GitHub Actions rail.

After this lesson you'll be able to trace the deployed request path and name the role of each AWS service in the first slice.

Concrete example: the browser asks the API for hello

In the current Dreamcatcher walking skeleton, the browser reaches one public thing: CloudFront. When the web app calls /api/hello, CloudFront routes that API path to an Application Load Balancer. The ALB is internal, not internet-facing, so CloudFront reaches it through a VPC origin. Behind the ALB is an ECS Fargate task running the Hono BFF container. AWS describes Fargate as a way to run ECS containers without managing EC2 instances or cluster capacity (AWS ECS Fargate docs).

This is deliberately a walking skeleton: not the whole product, but a thin slice proving that the delivery path, container runtime, edge, health checks, and smoke proof can all work together. The project evidence is in infra/terraform/production/main.tf, scripts/walking-skeleton/smoke.sh, and the BFF routes in apps/dreamcatcher-bff/src/app.ts.

Concrete example: the browser asks for static assets

The Vite bundle is not served by the BFF container. It is uploaded to a private S3 bucket. CloudFront reads that bucket using Origin Access Control, so the bucket can block public access while still serving viewers through CloudFront. AWS recommends OAC over the older OAI mechanism because it supports newer regions, SSE-KMS, and dynamic S3 requests (AWS CloudFront OAC docs).

That split matters: the API can roll as a container, while the static shell can be published with S3 sync and CloudFront invalidation. The smoke test checks both halves: API build metadata and a marker inside the web bundle.

The principle

The first AWS slice is a proving path, not a maximal architecture. It proves that a reviewed commit can become a public HTTPS experience with no public ALB, no long-lived GitHub AWS keys, Terraform-owned infrastructure, and a smoke test that names the exact commit being served. Everything missing from the slice is missing on purpose: RDS, real identity, workers, custom domains, WAF, and secrets arrive only when they can be proven end to end.

Service map

Service Job in this slice
CloudFront Only public entry point; routes static and API paths.
S3 assets bucket Private storage for the Vite bundle.
ALB Internal HTTP origin for the API container.
ECS Fargate Runs the API-only BFF as a container without managing EC2 nodes.
ECR Stores the BFF image tagged with the git SHA.
CloudWatch/SNS/SQS Minimal alarm path with a Terraform-owned SQS sink for proof.

Check yourself

  1. What is the only public entry point in the walking skeleton?
    Answer

    CloudFront. The ALB is internal, and the S3 assets bucket is private behind OAC.

  2. Why is the web bundle not served directly by the BFF container?
    Answer

    Static assets and API have different deployment and caching needs. S3 plus CloudFront handles the static bundle; ECS handles the API process.

  3. If a smoke test proves /api/build returns the expected commit SHA, what has it proven?
    Answer

    It proves the deployed API task is running the image built from that commit, not merely that some service is healthy.

  4. Which future feature would force this slice to grow: real customer data, custom domain, or another proof marker?
    Answer

    Real customer data. That introduces database, secrets, tenant isolation, backup, and restore concerns. A custom domain is also a future slice, but it is not required to prove the core rail.

  5. Explain the phrase "walking skeleton" in this project without naming any AWS service.
    Answer

    It is the thinnest production-shaped path that proves code can move from a reviewed commit to a working public system with observable proof, while leaving deeper product capabilities for later slices.
