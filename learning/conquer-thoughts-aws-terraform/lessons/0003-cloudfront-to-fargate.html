CloudFront to Fargate

Lesson 0003 - Conquer Thoughts AWS/Terraform - 2026-06-14

Part of your mission to build a durable mental model of the Conquer Thoughts production walking skeleton so you can explain, debug, and safely extend the AWS/Terraform/GitHub Actions rail.

After this lesson you'll be able to explain how CloudFront reaches private assets and an internal ALB without making the ALB public.

Concrete example: private S3, public website

The static web bundle lives in an S3 bucket with public access blocked, versioning enabled, and server-side encryption. CloudFront has an S3 origin configured with Origin Access Control. The bucket policy allows the CloudFront service principal to read objects only when the source ARN is this distribution. That is why the bucket can stay private while viewers still load the site through CloudFront.

CloudFront's OAC docs recommend signing origin requests, and the project sets signing_behavior = "always" and signing_protocol = "sigv4" in the static-edge module (AWS OAC docs).

Concrete example: private ALB, public API

The ALB is also not public. It is internal and sits in private subnets. CloudFront reaches it through a VPC origin. AWS says VPC origin resources can include Application Load Balancers, and when CloudFront creates a VPC origin it creates a service-managed security group named like CloudFront-VPCOrigins-Service-SG. The project discovers that security group and allows it to reach the ALB on port 80 (AWS VPC origin docs).

From the ALB inward, the web-service module creates a target group, listener, Fargate task definition, ECS service, and task security group. The task security group allows inbound traffic only from the ALB security group on the container port. Day one tasks have public IPs for egress, but that is not public ingress: the security group still decides who can connect to the task.

The principle

Public entry and public origin are different ideas. The project wants a public product URL, not a public ALB. CloudFront is the public entry; S3 and ALB are origins with narrowly granted access. That preserves a simple external shape while shrinking the attack surface behind it.

Why the route looks slightly odd

ChoiceTrade-off accepted nowFuture move
Internal ALB behind CloudFront VPC origin CloudFront/VPC-origin dependency ordering is more complex. Keep one public front door; add custom domain/WAF later.
Fargate tasks in public subnets with public IPs Open egress is accepted for day-one ECR pulls and runtime simplicity. Move tasks private with NAT or VPC endpoints when dependencies are known.
CloudFront default certificate/domain No branded domain yet. Add Route 53/ACM/custom domain as its own Terraform slice.

Check yourself

  1. What prevents a browser from reading the assets bucket directly?
    Answer

    The bucket blocks public access, and the bucket policy grants read access to CloudFront through OAC for this distribution.

  2. What makes the ALB "internal" but still useful to the public site?
    Answer

    The ALB is not internet-facing; CloudFront reaches it privately through a VPC origin.

  3. Why is a task public IP not the same as public ingress?
    Answer

    Public IP gives the task outbound internet reachability, but inbound traffic is still controlled by its security group.

  4. If someone proposes opening the ALB to 0.0.0.0/0 for easier debugging, what project rule should you recall?
    Answer

    CloudFront is the sole public ingress. Debug through CloudFront or automated in-VPC checks, not by opening the origin.

  5. Which earlier lesson connects to this one: Terraform state or walking skeleton scope?
    Answer

    Both. The network/edge shape is the walking skeleton's public proof path, and Terraform state records which root owns each piece.
