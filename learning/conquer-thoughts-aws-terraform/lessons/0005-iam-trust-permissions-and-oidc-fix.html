IAM Trust, Permissions, and the OIDC Fix

Lesson 0005 - Conquer Thoughts AWS/Terraform - 2026-06-14

Part of your mission to build a durable mental model of the Conquer Thoughts production walking skeleton so you can explain, debug, and safely extend the AWS/Terraform/GitHub Actions rail.

After this lesson you'll be able to diagnose why `sts:AssumeRoleWithWebIdentity` failed and why widening permissions would not fix it.

Concrete example: the failure happened before Terraform

The first GitHub deploy attempt repeated "Assuming role with OIDC" and then failed with Not authorized to perform sts:AssumeRoleWithWebIdentity. That means the workflow did not yet have AWS credentials. The failure was at the role entrance, before Terraform, ECR, S3, CloudFront, or ECS permissions mattered.

This is the most important distinction in the whole OIDC story: 

Trust policy:
  Who is allowed to enter the role?

Permissions policy:
  Once inside the role, what AWS actions may it perform?

Concrete example: the trust policy was too clever

The project briefly tried to check separate GitHub token conditions for repository and repository owner alongside the main-branch subject. The repair removed the extra owner/repo conditions and kept the narrow, documented shape that the workflow needed: 

"token.actions.githubusercontent.com:aud" = "sts.amazonaws.com"
"token.actions.githubusercontent.com:sub" = "repo:citypaul/conquer-thoughts:ref:refs/heads/main"

AWS's GitHub OIDC role guidance says the sub condition is the core limiter for specific GitHub organizations, repositories, and branches, and AWS validates that a GitHub OIDC trust does not omit it or leave it as only a wildcard (AWS GitHub OIDC role docs). GitHub's AWS guide also uses aud plus sub in its AWS trust examples and notes that workflows using environments have a different subject format (GitHub OIDC in AWS docs).

The principle

When AssumeRoleWithWebIdentity fails, do not widen the role's service permissions. The workflow never got inside the role. Fix the front-door trust so it matches the identity token the workflow actually emits, then rerun the failed deploy. Old failed runs do not heal after a role trust change because the OIDC step already exited.

Plan role versus deploy role

RoleTrust shapePermissions intent
Deploy role Exact main-branch subject. Apply production infrastructure, push/read images, publish assets, run smoke checks.
Plan role Defaults to the same main-branch subject until PR planning is deliberately designed. Read Terraform state and AWS account metadata; lock state while planning.

The plan role default is intentionally conservative. Pull request planning is useful, but it widens who can read AWS metadata and state. That should be a deliberate workflow design, not an accidental wildcard in the first production deploy.

Future subject changes to treat as security changes

Workflow changeOIDC consequenceReview question
Add GitHub environment Subject becomes environment-shaped. Are environment branch/tag protections configured and reflected in IAM trust?
Allow tags Subject can become tag-shaped. Which tags are allowed to deploy, and why?
Allow pull requests to plan Subject must include PR-safe patterns. What state and account metadata may PRs read?
Use reusable workflow Additional workflow identity claims may matter. Which workflow file is trusted to request credentials?

Check yourself

  1. A deploy log fails at AssumeRoleWithWebIdentity. What category of IAM problem is it?
    Answer

    A trust policy problem. The workflow has not entered the role yet.

  2. Why would adding s3:* or ecs:* not fix that failure?
    Answer

    Those are after-entry permissions. They matter only after STS has successfully returned temporary role credentials.

  3. What does the current deploy sub value lock down?
    Answer

    The repository and branch identity: citypaul/conquer-thoughts on refs/heads/main.

  4. Why is adding a GitHub Environment not a harmless YAML-only change?
    Answer

    It changes the OIDC subject shape. The IAM trust policy and environment protection rules must be reviewed together.

  5. What should be your first debug artifact: Terraform plan output or the role trust policy?
    Answer

    The role trust policy, because a failed role assumption means Terraform never got AWS credentials to plan or apply.

Go deeper: GitHub OIDC reference - read the subject-claim examples before changing workflow triggers, environments, tags, or PR credential behavior.
