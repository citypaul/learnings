Deployment Rail, Provenance, and Guardrails

Lesson 0006 - Conquer Thoughts AWS/Terraform - 2026-06-14

Part of your mission to build a durable mental model of the Conquer Thoughts production walking skeleton so you can explain, debug, and safely extend the AWS/Terraform/GitHub Actions rail.

After this lesson you'll be able to explain the deploy workflow from checked-out SHA to smoke-tested CloudFront URL.

Concrete example: manual dispatch is not arbitrary promotion

The deploy workflow runs on pushes to main and also supports manual dispatch with a full commit SHA. But the dispatch is still fenced: the SHA must be a full lowercase 40-character Git SHA and must be reachable from origin/main. The workflow detaches at that exact commit before building. This turns manual dispatch into a redeploy of known mainline code, not a side channel for unreviewed code.

The workflow also pins third-party actions to commit SHAs. That reduces supply-chain drift: the action code used by deployment is not silently whatever a mutable tag points to tomorrow.

Concrete example: the image carries its own receipt

The BFF Dockerfile accepts build args for BUILD_COMMIT_SHA, BUILD_CREATED_AT, and BUILD_IMAGE_TAG. The publish script builds an ARM64 image and tags it with the first 12 characters of the deploy SHA. The BFF exposes /api/build, and the smoke test asserts that the deployed response matches the expected commit and image tag.

The ECR repository is configured with immutable tags. AWS ECR documents that immutable tags cannot be overwritten; pushing an existing tag returns an error (AWS ECR tag immutability docs). That is useful because a git-SHA tag should identify one artifact, not drift over time.

The principle

The rail is designed so every important claim has proof. The commit check proves the source. The image tag and build metadata prove the container. Terraform's image_tag variable makes the deploy an apply, not a side-channel ECS update. The web bundle marker proves static publishing. The smoke test proves the public edge is serving the expected API and shell.

Guardrails to remember

GuardrailWhat it prevents
USE_AWS_PROFILE=false in GitHub Prevents scripts from looking for workstation credentials in CI.
Allowed AWS account check Prevents deploys into the wrong account.
Production/rehearsal shape checks Prevents a production apply with destructive rehearsal inputs.
GitHub Actions concurrency group Queues deploys so two applies do not race the same state.
ECS deployment circuit breaker Rolls back a failed rolling deployment to the previous completed deployment. AWS documents this rollback behavior for ECS services using the circuit breaker (AWS ECS circuit breaker docs).
Smoke checks Turns "it deployed" into "the expected commit is serving through CloudFront."

Static asset ordering

The API deploy goes through Terraform, but static assets are published by script. The current publish script syncs fingerprinted assets with long cache headers, syncs the rest with no-store, and creates a CloudFront invalidation. The smoke test then reads the web root, finds the JavaScript bundle, and checks that the bundle contains the deploy proof marker. That keeps the static side honest even though it is not Terraform state.

Check yourself

  1. Why does the workflow require a full 40-character SHA for manual dispatch?
    Answer

    It avoids ambiguous or moving references and forces the deploy to name an exact commit.

  2. What makes the deploy "a Terraform apply" instead of an ECS side channel?
    Answer

    The BFF image tag is passed to Terraform as image_tag, so Terraform creates the task definition for that artifact.

  3. Why are immutable ECR tags useful when the tag is a git SHA?
    Answer

    A SHA tag should identify one artifact. Immutability prevents accidentally replacing the image behind an already-used tag.

  4. Which script-level guardrail prevents GitHub from using a local AWS profile?
    Answer

    Each walking-skeleton script refuses GITHUB_ACTIONS=true with USE_AWS_PROFILE=true.

  5. If the API smoke passes but the web marker fails, where would you look first?
    Answer

    Look at the web build, S3 sync, cache headers, CloudFront invalidation, and whether the expected bundle was actually referenced by the root HTML.
